🎨 Atlas UI Framework
The Atlas UI system is a custom, immediate-mode, GPU-accelerated UI toolkit built for sci-fi cockpit interfaces — and designed to be reusable in other projects.

→ Full Atlas UI Documentation

Key Features
Immediate-mode API — no retained widget trees; simple if (button(...)) pattern
Single draw-call batching — all UI rendered in one GPU pass
Sci-fi widget set — panels, status arcs, capacitor rings, module racks, overview tables
Interactive — drag-to-move panels, click buttons, tab switching, scrolling
Themeable — full color scheme support (default teal, classic amber, colorblind-safe)
Zero dependencies beyond OpenGL 3.3
Quick Example
#include "ui/atlas/atlas_context.h"
#include "ui/atlas/atlas_widgets.h"

atlas::AtlasContext ctx;
ctx.init();

// Each frame:
atlas::InputState input = getInputFromGLFW();
ctx.beginFrame(input);

atlas::Rect panelBounds = {100, 100, 300, 200};
if (atlas::panelBegin(ctx, "My Panel", panelBounds)) {
    if (atlas::button(ctx, "Click Me", {110, 140, 80, 24})) {
        // handle click
    }
    atlas::progressBar(ctx, {110, 170, 200, 16}, 0.75f,
                       ctx.theme().shield, "Shield: 75%");
}
atlas::panelEnd(ctx);

ctx.endFrame();

Good test. That problem is very common when mixing ECS transforms + world axes + camera rotation.
What you’re seeing means the FPS controller is using world-space movement instead of camera-relative movement, so W/A/S/D moves along global X/Z instead of the direction the player is looking.

This is not a rendering problem.
It’s an input → movement → transform math bug in the FPS controller.

We need to fix the FPS movement system so it behaves like a real FPS inside stations / ships / EVA.

🎯 Problem
Current behavior:

W = +Z world
S = -Z world
A = -X world
D = +X world
Correct FPS behavior:

W = forward relative to camera
S = backward relative to camera
A = left relative to camera
D = right relative to camera
So movement must use camera basis vectors, not world axes.

✅ Correct FPS Movement Model
You need:

camera.forward
camera.right
camera.up
Then movement:

move = forward * inputY + right * inputX
NOT:

move = vec3(inputX, 0, inputY)

Pseudo C++ (Atlas style)

🧱 Common mistake in ECS FPS controllers
Bad:

Correct:

vec3 f = getCameraForward();
vec3 r = getCameraRight();

transform.pos += f * inputY;
transform.pos += r * inputX;
